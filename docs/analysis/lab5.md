# Лабораторна 5: Модульний Моноліт

## Частина 1. Аналіз Лабораторної 5: Модульний Моноліт

### 1. Як визначили межі контекстів? Які альтернативи розглядали?

Систему було розділено на три обмежені контексти (Bounded Contexts):

1. **Core Context (Venue + Registrations):** Управління локаціями та реєстраціями користувачів. Вони об'єднані, оскільки реєстрація завжди відбувається на конкретному venue, і розділяти їх було б штучним розривом єдиної бізнес-транзакції.
2. **Events & Tickets Context:** Управління подіями та типами квитків. Взаємодіє з Core через чітко визначені публічні API.
3. **Analytics Context (новий):** Збір метрик та побудова проєкцій. Це ізольований read-only споживач, який працює через події.

**Альтернативи, які розглядалися:**

- _Venue як окремий контекст:_ Відхилено. На практиці Venue існує виключно для забезпечення подій та реєстрацій. Виділення його в окремий модуль створило б зайві кордони та ускладнило комунікацію там, де потрібна атомарність.
- _Весь бекенд як один великий контекст:_ Відхилено. Хоча це зменшує overhead на комунікацію, це повністю нівелює переваги модульного моноліту (ізоляцію, підготовку до мікросервісів) та не дозволяє застосувати патерни ACL і Eventual Consistency.

### 2. Як ACL захищає модулі від змін в інших модулях?

**Anti-Corruption Layer (ACL)** виступає захисним бар'єром, який перекладає зовнішню модель даних одного модуля у внутрішню модель модуля-споживача.

_Приклад із проєкту (Registrations → Venue):_ Модулю `Registrations` потрібні дані про локацію для створення запису. Без ACL модуль реєстрацій напряму залежав би від структури DTO модуля `Venue`. Якщо у `Venue` вирішать розділити поле `address` на `city` та `street`, модуль `Registrations` зламається.
Завдяки ACL модуль `Registrations` має власний адаптер (`VenueAclTranslator`), який звертається до публічного API `Venue`, отримує їхній DTO і збирає з нього власну внутрішню модель `VenueInfo`. Будь-які зміни в базі чи контрактах `Venue` вимагатимуть лише оновлення адаптера-транслятора, тоді як основна бізнес-логіка `Registrations` залишиться недоторканою.

### 3. Де у системі strong consistency, а де eventual consistency? Чому це прийнятно?

**Strong Consistency (Сильна консистентність):**
Використовується в межах одного контексту або для критичних синхронних перевірок.

- _Приклади:_ 1) Видалення локації (`Venue`). Перед видаленням система робить синхронний запит до модуля `Events`, щоб перевірити наявність запланованих подій. 2) Коли користувач реєструється на подію — в одній транзакції перевіряється ліміт квитків і зберігається реєстрація. Якщо ліміт вичерпано — операція відхиляється.
- _Чому прийнятно:_ Для ключових бізнес-правил затримка в узгодженості даних є неприпустимою. Строга консистентність вимагає транзакційних блокувань у базі даних. Оскільки ми застосовуємо її лише локально (всередині одного модуля), локальні транзакції не блокують роботу інших незалежних частин системи.

**Eventual Consistency (Кінцева узгодженість):**
Використовується для міжмодульної комунікації через шину подій (EventBus).

- _Приклад:_ `Core` модуль публікує `VenueCreatedEvent` або `RegistrationCreatedEvent`. `Analytics` та `Notifications` обробляють ці події асинхронно у фоні.
- _Чому прийнятно:_ Збій сервісу нотифікації не повинен блокувати критичну бізнес-операцію (реєстрацію юзера), а оновлена статистика, яка прийшла на секунду пізніше, не повинна впливати на цілісність системи.

### 4. Що зміниться, якщо потрібно буде виділити один з модулів у окремий сервіс?

Якщо ми вирішимо винести `Analytics` в окремий мікросервіс, бізнес-логіка та обробники подій залишаться без змін завдяки вже впровадженим кордонам.
Основні інфраструктурні зміни:

1. **Event Bus:** Замість `InProcessEventBus` доведеться розгорнути Message Broker (наприклад, RabbitMQ або Kafka) для передачі подій між фізичними серверами.
2. **База даних:** Спільна PostgreSQL БД розділиться на дві незалежні.
3. **Комунікація:** Синхронні виклики доведеться замінити на мережеві запити (HTTP/REST або gRPC).
4. **Відмовостійкість:** З'явиться потреба в ідемпотентності, механізмах Retry та Dead Letter Queue для обробки мережевих збоїв.

## Частина 2. Ретроспектива курсу

### 1. Як змінювалася архітектура проєкту на кожному етапі?

- **Лабораторна 1 (Базовий бекенд):** Проста MVC-подібна структура (Controller → Service → Repository). Була присутня висока зв'язність, бізнес-логіка змішувалася з SQL-запитами, а тестування без бази даних було неможливим.
- **Лабораторна 2 (Layered Architecture + DDD):** Впровадження інверсії залежностей та багатошаровості. З'явилася Rich Domain Model та фабрики. Бізнес-логіка стала ізольованою та придатною для unit-тестування.
- **Лабораторна 3 (CQS):** Розділення потоків на читання (Queries) та запис (Commands). Це дозволило оптимізувати вибірки даних (прямі SQL без мапінгу в домен) та застосувати принцип єдиної відповідальності (SRP) для хендлерів.
- **Лабораторна 4 (Асинхронна комунікація):** Впровадження In-Process Event Bus та доменних подій. Система стала більш стійкою, відкладаючи побічні ефекти (відправку нотифікацій) у фонові процеси.
- **Лабораторна 5 (Модульний моноліт):** Формування Bounded Contexts. Впровадження публічних API для кожного модуля, ACL для захисту кордонів та Integration Events для eventual consistency між модулями.

### 2. Які архітектурні рішення виявилися найціннішими?

1. **Dependency Inversion Principle:** Дозволив легко мокати інфраструктуру і тестувати ядро системи в ізоляції.
2. **CQS:** Розділення моделей читання і запису кардинально спростило написання складних запитів до БД без перевантаження доменних сутностей.
3. **Integration Events та ACL:** Дозволили створити повноцінні незалежні модулі, зміна в яких не тягне за собою каскадне руйнування всього проєкту.

### 3. Що було б зроблено інакше, знаючи кінцевий результат?

- **Поступове впровадження DDD:** У лабораторній 2 з'явилося одразу дуже багато boilerplate-коду (фабрики, мапери, DTO). Знаючи кінцеву мету, можна було б стартувати з простіших доменних моделей і ускладнювати їх лише там, де реально виникала складна бізнес-логіка.
- **Чітке розмежування подій:** Варто було ще на етапі 4-ї лабораторної явно розділити події на _Domain Events_ (внутрішні для модуля) та _Integration Events_ (публічні контракти). Це зробило б перехід до модульного моноліту у 5-й лабораторній більш безшовним.

### 4. Які trade-offs побачили між простотою та гнучкістю?

- **Швидкість розробки vs Архітектурний overhead:** Простий CRUD з 1-ї лаби дозволяє швидко доставляти фічі, але швидко перетворюється на спагетті-код. DDD та CQS генерують багато файлів на одну дію (Command, Handler, Event, Mapper), але ця складність окупається підтримкою на довгостроковій дистанції.
- **Strong vs Eventual Consistency:** Сильна консистентність робить систему передбачуваною і простою для розуміння, але створює вузькі місця у продуктивності. Кінцева узгодженість (eventual consistency) значно пришвидшує відповідь сервера для клієнта, але переносить складність на рівень обробки помилок (необхідність retry-логіки та відстеження "загублених" подій).
- **Модульний моноліт vs Мікросервіси:** Модульний моноліт є ідеальним компромісом. Він дає гнучкість і незалежність мікросервісів, але позбавляє від операційного пекла (network latency, distributed tracing, orchestration), дозволяючи винести модулі в окремі сервіси лише тоді, коли це справді стане необхідно.
