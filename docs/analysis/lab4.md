# Лабораторна робота 4: Аналіз синхронної та асинхронної комунікації

## Що змінилося порівняно з лабораторною 3?

У лабораторній 3 вся логіка виконувалася лінійно: controller → handler → domain → repository.
Жодних побічних ефектів — handler відповідав виключно за основну бізнес-операцію.

У лабораторній 4 з'явився новий модуль — `notifications`, який відповідає за побічні операції.
Були реалізовані два способи комунікації з ним: синхронний (прямий виклик через інтерфейс)
та асинхронний (публікація події через Event Bus).

Структурно додалося:

- `src/shared/domain/events/` — Integration Events (`RegistrationCreatedEvent`, `EventCancelledEvent`)
- `src/shared/application/ports/event-bus.interface.ts` — контракт Event Bus
- `src/shared/infrastructure/event-bus/in-process-event-bus.ts` — реалізація Event Bus
- `src/modules/notifications/` — новий модуль з власним контрактом і реалізацією

## Які побічні операції виділені і чому?

Виділено дві побічні операції:

**1. Підтвердження реєстрації** (`CreateRegistration` → `RegistrationCreatedEvent`)

Коли користувач реєструється на подію — логічно надіслати підтвердження.
Ця операція є побічною: реєстрація вже збережена в БД незалежно від того,
чи дійде нотифікація. Збій нотифікації не повинен скасовувати реєстрацію.

**2. Сповіщення про скасування події** (`CancelEvent` → `EventCancelledEvent`)

Коли організатор скасовує подію — учасники мають бути повідомлені.
Це класичний кейс для демонстрації різниці між підходами: при синхронному варіанті
організатор чекає поки система "сповістить" (навіть якщо це просто console.log),
при асинхронному — отримує відповідь одразу.

Обидві операції мають спільну характеристику: вони не є частиною основної бізнес-транзакції
і можуть бути відкладені без шкоди для консистентності даних.

## Порівняння підходів

### Час відповіді API

**Синхронний варіант:**
Час відповіді = час основної операції + час нотифікації.
У поточній реалізації нотифікація — це `console.log`, тому різниця мінімальна.
В реальній системі (HTTP-запит до email-сервісу, запис у чергу) час відповіді
зростає пропорційно кількості та складності побічних операцій.

**Асинхронний варіант:**
Час відповіді = час основної операції + час публікації події в Event Bus.
Публікація в in-process Event Bus — це виклик `Promise.all` на список handlers,
тобто фактично теж синхронний в межах одного процесу.
Справжня перевага проявляється при зовнішньому брокері (RabbitMQ, Redis):
handler публікує подію в чергу і повертає відповідь клієнту, а підписники
обробляють її незалежно і пізніше.

### Поведінка при збоях

**Синхронний варіант:**
Збій нотифікації перехоплюється через `try/catch` після збереження основних даних.
Помилка логується, але не пробрасується — основна операція вже завершена успішно.
Це свідоме рішення: нотифікація є побічною операцією і не повинна відкочувати
збережену реєстрацію чи скасування події.

Якби `try/catch` не було — збій нотифікації повертав би 500 клієнту,
хоча дані вже збережені в БД. Це було б некоректно.

**Асинхронний варіант:**
Handler публікує подію і завершується. Якщо підписник впаде — основна операція
вже давно завершена і клієнт отримав відповідь. Збій ізольований у підписнику.

При in-process Event Bus збій підписника все ще відбувається в тому ж процесі,
але після відповіді клієнту. При зовнішньому брокері — підписник може впасти
і перезапуститися незалежно, повторно обробивши подію з черги.

### Зв'язаність між компонентами

**Синхронний варіант:**
`CreateRegistrationCommandHandler` залежить від `INotificationService`.
Handler знає що існує нотифікація і явно її викликає.
Додати нову побічну операцію = додати нову залежність у конструктор handler-а
і новий виклик у тілі методу.

**Асинхронний варіант:**
`CreateRegistrationAsyncCommandHandler` залежить від `IEventBus`.
Handler не знає про нотифікації — він публікує факт (`RegistrationCreatedEvent`)
і не знає хто на нього підписаний.
Додати нову побічну операцію = створити нового підписника і зареєструвати його
в `app.ts`. Handler не змінюється.

### Складність реалізації та тестування

**Синхронний варіант:**
Простіший у реалізації — прямий виклик через інтерфейс.
Легко тестується: мокуємо `INotificationService` і перевіряємо що метод викликався
з правильними аргументами. Перевірка поведінки при збої — кидаємо помилку з мока
і перевіряємо що основна операція все одно успішна.

**Асинхронний варіант:**
Складніший: потрібен Event Bus, підписники, реєстрація в Composition Root.
Тестування розділяється на два рівні:

- Handler тестується ізольовано: мокуємо `IEventBus`, перевіряємо що подія опублікована
- Notification handler тестується окремо: передаємо подію напряму, перевіряємо реакцію

## Який підхід обрати для production-системи?

Для даного домену — **гібридний підхід**, залежно від конкретної операції:

**Синхронно** — якщо результат побічної операції потрібен для відповіді клієнту,
або якщо збій побічної операції має скасувати основну (наприклад, резервування місця
на події разом з оплатою).

**Асинхронно** — для нотифікацій, аудиту, аналітики.
Ці операції можуть бути відкладені без шкоди для користувача.
Нотифікація, що прийшла на секунду пізніше — прийнятна.
Нотифікація, що заблокувала реєстрацію — неприйнятна.

Для production `ConsoleNotificationService` замінюється реальною реалізацією
(email через SendGrid, SMS через Twilio тощо) без змін у жодному handler-і —
завдяки DIP.

При зростанні навантаження in-process Event Bus замінюється на RabbitMQ або Redis Streams —
також без змін у handler-ах, лише нова реалізація `IEventBus`.

## Ідемпотентність обробників

**Поточна реалізація** (`ConsoleNotificationService`) є природно ідемпотентною —
повторний `console.log` не має побічних ефектів.

**В реальній системі** повторна доставка події (at-least-once delivery при зовнішньому брокері)
призведе до дублювання нотифікацій — користувач отримає два email.

Рішення — дедуплікація за `registrationId` і типом нотифікації:

```typescript
if (await this.notificationRepo.alreadySent(event.registrationId, "confirmation")) {
  return;
}
await this.notificationService.sendRegistrationConfirmation(payload);
await this.notificationRepo.markAsSent(event.registrationId, "confirmation");
```

При in-process Event Bus з однією БД повторна доставка малоймовірна,
тому повноцінна дедуплікація для поточної реалізації є надлишковою.
Вона стає необхідною при переході на зовнішній брокер з retry-логікою.
