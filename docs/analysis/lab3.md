# Event Service

Event Service — платформа, яка об'єднує організаторів заходів та покупців. Сервіс дозволяє адміністраторам створювати події, додавати локації (venues) та налаштовувати різні типи квитків із зазначенням лімітів та цін, а звичайні користувачі можуть переглядати події та купувати квитки, що фіксується у системі як реєстрація на певну подію.

---

## Стек технологій:

- Мова програмування — TypeScript;
- Framework — Fastify;
- ORM — TypeORM;
- СУБД — PostgreSQL;
- Контейнеризація — Docker, Docker Compose.

## Переваги CQS: чим допомогло розділення?

- **Оптимізація без компромісів:** Операції запису (команди) вимагають надійності, транзакцій та суворої перевірки доменних інваріантів. Операції читання (запити) вимагають швидкості та зручної структури даних для клієнта. Завдяки CQS можна оптимізувати читання (читати безпосередньо з БД без підняття важких доменних об'єктів), не порушуючи надійності запису.
- **Безпечність:** Виклик Query ніколи не мутує стан системи, тому його завжди безпечно викликати повторно, кешувати чи масштабувати.

---

## Недоліки CQS: що ускладнилося?

- Замість одного методу в сервісі тепер потрібно створювати мінімум три нові сутності: об'єкт Command/Query (DTO), сам Handler та, за необхідності, нові методи в інтерфейсах репозиторіїв. Це збільшує кількість написаного коду.
- Ускладнюється навігація по проєкту: логіка, яка раніше була згрупована в одному файлі, тепер розкидана по декількох дрібних файлів-обробників.
- Зростає кількість маппінгів між різними моделями даних (HTTP Request → Command/Query DTO, ORM Entity → Read DTO).

## Чим підхід з Command / Query Handler відрізняється від Service, який робить все сам?

- **Service (Єдиний сервіс):** Об'єднує всі методи в одному великому класі, що призводить до перевантаження залежностями. Зміна однієї операції несе ризик зламати інші, а для тестування навіть найпростішого методу доводиться мокати всі залежності всього сервісу.
- **Command / Query Handlers:** Кожна операція виділяється у власний невеликий клас із єдиною відповідальністю (SRP). Handler отримує лише ті залежності, які необхідні конкретно йому, що робить його ізольованим, стабільним і максимально простим для тестування.

---

## Як CQS впливає на розширюваність? (додати нову команду vs змінити існуючий сервіс)

CQS ідеально підтримує архітектурний принцип Open/Closed Principle. У підході з єдиним великим сервісом, щоб додати нову функцію, доводиться змінювати існуючий клас і, можливо, розширювати його залежності, що створює ризик випадково зламати роботу сусідніх методів та змушує переписувати наявні тести всього сервісу. Натомість, в архітектурі CQS для додавання нової операції достатньо просто створити абсолютно новий об'єкт Command (DTO) та відповідний клас CommandHandler. Це дозволяє додавати новий функціонал без жодного втручання у вже існуючий та протестований код, роблячи систему максимально безпечною та легкою до масштабування.

---

## Чи відрізняється структура даних, яку повертає Query, від доменної моделі? Чому це важливо?

Так, вони кардинально відрізняються, оскільки запити повертають Read Models або DTO, а не безпосередньо доменні моделі. Доменна модель інкапсулює бізнес-логіку та складні інваріанти, тоді як Read Model — це проста структура без поведінки, оптимізована під зручність клієнта або UI (часто містить денормалізовані дані для уникнення зайвих запитів до БД). Таке розділення є критично важливим для уникнення жорсткої архітектурної прив'язки зовнішнього АРІ до внутрішньої логіки. Використання DTO створює чіткий захисний кордон, що дозволяє доменному шару та зовнішньому інтерфейсу функціонувати абсолютно незалежно один від одного.
